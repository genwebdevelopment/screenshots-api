Screenshot API — How We Use It

A standalone Node + Puppeteer HTTP service that captures website screenshots on demand. It runs on a Hostinger VPS and is called by the agents-dashboard WordPress plugin, which is hosted on Kinsta where shell_exec (and therefore a local headless Chrome) is blocked. This service moves the heavy, browser-driven work off WordPress and onto a box that can run Chromium.

Why this service exists

  • Kinsta can't run Chrome. The WP plugin needs full-page and section screenshots, but Kinsta's managed environment blocks shell_exec, so Puppeteer/Chromium can't launch there.
  • Offload to a VPS. This API runs Chromium on a Hostinger VPS (≥2 GB RAM) and exposes a simple authenticated HTTP endpoint.
  • Reuse & speed. A single browser instance is kept alive across requests, and concurrency is capped so the box doesn't run out of memory (~500 MB per concurrent Chrome).

How a request flows

  1. The WP plugin sends POST /screenshot with the target url, a list of sections, and the x-api-key header.
  2. The request is queued through a concurrency limiter (MAX_CONCURRENCY, default 2).
  3. Puppeteer opens the page, waits waitTime ms for content to settle, then captures each requested section.
  4. PNGs are written to disk under SCREENSHOT_DIR/<jobId>/ and a job record is saved to the JSON store.
  5. The response returns the jobId and a list of public file URLs (served from /files/<jobId>/<filename>).

Endpoints

GET /health

Unauthenticated health check. Returns { ok: true, queued: N } where queued is active + pending capture jobs.

POST /screenshot

Requires the x-api-key header (or ?api_key= query param). Request body:

{
  "url": "https://example.com",
  "sections": ["full", "hero", "slider", "map", "footer"],
  "viewport": { "width": 1920, "height": 1080 },
  "waitTime": 3000,
  "timeout": 60000
}

Response:

{
  "jobId": "abc123",
  "files": [
    {
      "section": "full",
      "filename": "full-page.png",
      "url": "https://host/files/abc123/full-page.png"
    }
  ]
}

Notes: url must start with http:// or https://. waitTime is clamped to 0–15000 ms and timeout to 5000–120000 ms. If no sections are given, it defaults to ["full"]. A job is recorded even on failure so it shows up in the dashboard.

GET /files/:jobId/:filename

Static access to the captured PNGs (cached for 1 day).

Dashboard API (all require x-api-key)

  • GET /api/stats — totals, today/week counts, failures, avg duration, disk usage.
  • GET /api/jobs?limit=&offset=&status= — paginated job history.
  • GET /api/jobs/:id — a single job record.
  • DELETE /api/jobs/:id — delete a job and its files.
  • POST /api/jobs/cleanup — body { "olderThanDays": 7 }, prunes old jobs.
  • DELETE /api/jobs — delete ALL jobs and files.

Sections we capture

The sections array picks what to grab. Each maps to a capture strategy in src/screenshot.js:

SectionWhat it capturesHow
fullWhole pageFull-page screenshot
heroTop hero areaViewport clip from y=0, 900px tall
sliderServices sliderElement: #find-services, .services-sec, .looking-for-today-slider
mapUS mapElement: #vmap, .us-map, [class*="map-section"], .state-map
footerPage footerElement: footer, .elementor-location-footer, [data-elementor-type="footer"]

For element sections, the service tries each selector in order and uses the first match. If nothing matches, that section is reported as element_not_found but the rest of the job still runs.

Example call

curl -X POST https://your-domain/screenshot \
  -H "Content-Type: application/json" \
  -H "x-api-key: YOUR_KEY" \
  -d '{"url":"https://example.com","sections":["full","hero","footer"]}'

Deployment (Hostinger VPS)

  1. Use a VPS plan with ≥2 GB RAM (Chrome is memory-hungry).
  2. Install Node 20+ and the Chromium system dependencies (see README.md for the full apt list).
  3. Upload this folder, copy .env.example → .env, and set API_KEY, PUBLIC_BASE_URL, PORT.
  4. Run npm ci --omit=dev, then start under PM2: pm2 start ecosystem.config.js && pm2 save && pm2 startup.
  5. Put nginx + Let's Encrypt in front for HTTPS and rate limiting (recommended).

Operational notes

  • The browser instance is reused across requests for speed; raise MAX_CONCURRENCY only if RAM allows.
  • Screenshots persist on disk — use the dashboard Cleanup buttons or a cron job to prune old jobs and keep disk usage down.
  • Every request (success or failure) is logged to the job store and visible in the Jobs tab, with thumbnails, duration, size, and client IP.
